# Scope Review — install-rewrite

## Critical Gaps (must answer before building)

- **Theme baseline (`config/`) is scope-crept.** BACKLOG.md:46-68 says nothing about shipping a `cmux.json`, `tmux.conf`, or `zsh-prompt-colors.zsh`. The backlog item is exclusively about *detect → install → verify → onboard* for tools the pipeline uses. The whole "MonsterFlow shell theme" stage (Spec lines 40-42, 134-137, 204-213, 249-255, plus acceptance case 6 at line 344, plus Open Question 3 at line 356) is a new feature that the user did not ask for in the backlog. Decision needed before /build: **either drop the theme stage entirely**, or **promote it to its own spec** (`docs/specs/shell-theme/`) so it can be reviewed on its own merits. Keeping it inside install-rewrite roughly doubles the diff and adds a whole new prompt-flow + sentinel-block + uninstall question.
- **`cmux` cask added to RECOMMENDED is also scope creep.** BACKLOG.md:63 only names `git/python/gh/shellcheck/jq/tmux`. Spec line 39 demotes tmux and adds cmux as RECOMMENDED. cmux is not used by any pipeline command in the repo today (verify before adding). If cmux is added because the *theme* needs it, that's a second-order scope creep cascading off the first. Decision needed: justify cmux in RECOMMENDED with a concrete pipeline dependency, or move it to OPTIONAL (or drop).
- **Wiki indexing kickoff is scope-crept.** Backlog line 57 says "Optionally offer to run `bootstrap-graphify.sh` against `~/Projects/`" — graphify only. Spec line 44 adds wiki indexing (`wiki-export` skill on `~/.obsidian-wiki/config` detection) and acceptance case 7 (line 345) tests it. Wiki is a Justin-specific dependency (Obsidian vault); no adopter will have `~/.obsidian-wiki/config`, so the gate is dead code for everyone but the owner. Decision needed: drop wiki kickoff from install-rewrite (it's an owner-only convenience, not adopter onboarding) and keep graphify-only per backlog.

## Important Considerations

- **`config/` file contents are an unbounded balloon.** Open Question 3 (line 356) admits the spec doesn't pin the contents of any of the three theme files and defers to /plan. This is exactly the gold-plating risk the scope-review checklist flags ("'while we're in there' refactors"). A `tmux.conf` ported from Justin's personal config plus a `cmux.json` plus a sourceable zsh-colors file is easily 100-300 lines of opinionated keybindings and color tuning — none of which closes the BACKLOG.md gap (an adopter who ran `./install.sh` and got a working symlink graph but no `gh` is no less stuck if you also hand them cyan tmux panes). If theme stays in scope despite the Critical Gap above, cap each file at ~30 lines and ban anything beyond colors + one source line.
- **"Out of scope" list is missing several near-certain follow-ups.** Spec lines 51-55 list 5 deferrals but omits: (a) custom user override of theme colors (will be the first adopter ask the day after launch — "I want solarized, not cyan"), (b) uninstall-theme path (spec line 321 already hand-waves this to a future `scripts/uninstall-theme.sh`), (c) per-tool opt-in within RECOMMENDED ("install gh but not cmux"), (d) Linux/WSL adopters who hit the brew error and want a fallback. Adding these to "Out of scope" now prevents litigation during /build.
- **Open Questions 1, 2, 4 are correctly out-of-scope; #3 is in-scope-disguised-as-deferred.** Q3 ("`config/` file contents") is not really an open question — it's a missing requirement. /plan cannot resolve "what colors should tmux use?" without product input. Either pin contents in this spec, or drop the theme stage. Don't ship "/plan figures it out" as a real answer; that's where scope creeps in unobserved.
- **Acceptance case count vs. value.** 7 cases total. Cases 1-5 directly map to BACKLOG.md:60-64 idempotency hard rules (perfect alignment). Case 6 (theme owner-vs-adopter default) and Case 7 (indexing kickoff gates) test scope-crept features. If theme + wiki indexing exit the spec, the test surface drops from 7 to 5 — which is the right MVP. Each acceptance case is a build/maintain cost; right-sizing here pays compounding interest.
- **`python_pip()` migration is correctly deferred to follow-up** (Open Question 2, line 355). Good discipline. Keep it deferred.

## Observations

- **MVP cut would deliver the BACKLOG.md gap in roughly half the spec.** A minimum viable install-rewrite is: (a) parse `--no-install` flag, (b) extend tier detection to include `brew` in REQUIRED, (c) run `brew bundle --file=Brewfile` after detection (single confirm), (d) tier-split decline behavior, (e) `scripts/onboard.sh` that calls `doctor.sh` + prints next-steps panel + offers graphify bootstrap + offers `gh auth login` + prints codex one-liner, (f) `tests/test-install.sh` cases 1-5. That's stages 1, 3, 4, 12 from the spec's flow diagram — scope-creep stages (7 theme, plus theme-related parts of stage 12) drop out. Diff stays focused on the actual adopter-drop-off problem.
- **Additive-surgery vs. wholesale-rewrite call is sound** (Approach section, lines 58-83). The PWD-vs-basename memory entry, sentinel-bracketed gitignore block, and persona-metrics default-flip are all hard-won; preserving them verbatim is the right risk posture. This part of the spec is well-scoped.
- **"Owner-vs-adopter default flip" pattern is reused well** for the theme stage if theme stays — leverages an existing-and-proven mechanism rather than inventing a new one. Same pattern for the `.zshrc` sentinel block (lines 253-255). Good reuse.
- **Stage 11 "Validate" runs the full `tests/run-tests.sh`.** Worth confirming this isn't already covered elsewhere — if `install.sh` runs tests every time, that's a slow path on owner re-runs and may collide with the "<2 seconds" acceptance bar (case 2, line 340). Quick check during /plan.
- **The flow diagram (lines 64-76) shows 12 stages where 8 existed before** — a 50% surface increase. New stages: 1 (flag parsing), 3 (install), 4 (decline), 7 (theme), 12 (onboard). If theme exits scope, the increase is 4 stages — proportional to the backlog ask. If theme stays, the spec is doing two things at once.

## Verdict

**FAIL** — The theme baseline (`config/` directory + `--install-theme`/`--no-theme` flags + `.zshrc` sentinel block + acceptance case 6) and the wiki-indexing kickoff (acceptance case 7) are net-new features absent from BACKLOG.md:46-68 and should be split into a separate spec or dropped before /plan; the install-rewrite spec should focus exclusively on the detect → install → verify → onboard pivot the backlog actually asked for.
