# Stakeholders Review — install-rewrite

## Critical Gaps (must answer before building)

- **Linux-user fail mode is undefined.** Spec marks Linux out-of-scope but the script will still run if a Linux user clones and executes it. There is no early-exit guard like `[[ "$(uname)" == "Darwin" ]] || { echo "macOS only — see issue #X"; exit 2; }`. A Linux adopter today gets a cryptic `command not found: brew` after the REQUIRED panel, with no signpost that this is expected and no link to a tracking issue. This is a stakeholder the spec acknowledges exists but actively under-serves. Add an explicit OS guard to the flag-parse stage and a one-line message naming the platform restriction.
- **Existing v0.4.x adopter migration path is unaddressed.** Anyone who ran the previous install.sh has symlinks, possibly a persona-metrics gitignore block, possibly nothing in `~/.zshrc`, and definitely no `config/` directory. The spec's "re-run on fully-installed system" edge case assumes the new code is what created the system — it doesn't describe what an old-install adopter sees on their first upgrade run (will they get the theme prompt unexpectedly? Will brew-bundle fire with a long install list because they never had `gh`/`shellcheck`/`jq`?). At minimum the spec needs an "upgrade UX from v0.4.x" subsection in UX/User Flow plus a corresponding acceptance test (case 8: pre-stage a v0.4.x-shaped `$HOME`, run new install.sh, assert no surprise mutations).
- **`gh auth login` non-interactive failure mode.** onboard.sh calls `gh auth status` and offers `gh auth login` if unauthenticated. But adopters using `GH_TOKEN` env var or `gh auth login --with-token` from a script will register as "unauthenticated" via `gh auth status` in some configurations, and the `gh auth login` prompt is a fully interactive TTY flow that will hang or error in any non-TTY context (CI, cmux automation, ssh without `-t`). Spec must either (a) detect non-TTY and skip the offer silently, or (b) detect `GH_TOKEN`/`GITHUB_TOKEN` env and skip the offer. Today's spec will hang an autorun.

## Important Considerations

- **Owner-debug session theme mutation risk.** Owner-default-YES on theme means every time Justin runs `./install.sh` from `$REPO_DIR` to debug install behavior, the script will (re-)symlink `~/.tmux.conf`, `~/.config/cmux/cmux.json`, and append the sentinel block to `~/.zshrc`. Idempotent, yes — but if Justin is mid-experiment with a custom tmux.conf or testing a different prompt, the install will silently overwrite his working state. Recommend: if the target file exists and is NOT already a symlink to the repo's config file, prompt before clobbering even in owner mode (or print "leaving existing ~/.tmux.conf alone — it's not our symlink").
- **`--no-install` is the only granular flag.** Spec offers `--no-install` (skip brew), `--install-theme`/`--no-theme` (theme control), but no `--no-onboard` (skip the onboard.sh call entirely) and no `--quiet` (suppress the boxed panel for scripted runs). A CI pipeline that wants symlinks-only with no panel output has no clean flag combination. Add `--no-onboard` for fully-scripted runs; `--quiet` is nice-to-have.
- **Adopter trust model is implicit, not proven.** Spec says "every byte is auditable before it lands on the adopter's disk" for the `config/` files but doesn't tell the adopter HOW to audit (e.g., "before running install.sh, read `config/zsh-prompt-colors.zsh` — it's 30 lines, no network calls, no eval"). The README/QUICKSTART update mentioned in the non-test acceptance bar should explicitly invite the audit and name the files. Adopters installing brew packages + writing to `.zshrc` deserve a one-paragraph "what this script will and will not do to your machine" upfront.
- **Theme uninstall is deferred to "if/when an adopter asks."** The spec deliberately doesn't ship `scripts/uninstall-theme.sh`. But the sentinel-bracketed `.zshrc` block already supports clean removal — so the uninstall is mostly a 10-line script that `sed`s out the block and `rm`s the two symlinks. Shipping it now (rather than waiting) closes a stakeholder loop: the adopter who tries the theme, decides they don't like it, and wants to back out cleanly. Without it, they'll edit `.zshrc` by hand and possibly mis-delete the sentinel markers.
- **Onboarding panel "indexing kickoff" stakeholder split.** onboard.sh offers `bootstrap-graphify.sh` if `~/Projects/` exists. For an adopter whose `~/Projects/` is huge (50+ repos, GBs of code), this kickoff is a heavy first impression — graphify is not free, even AST-only. Spec should clarify whether the prompt names the cost ("This will index N projects, takes ~M minutes") or just blindly offers it.

## Observations

- **Box-drawing degradation noted but not tested.** Edge case section mentions Unicode box-drawing degrades acceptably on legacy terminals but no acceptance case validates this. Low-stakes, but the support-team-question lens ("why is my onboard panel garbled?") suggests an `--ascii` flag or env-var fallback.
- **`gh auth login` is interactive-only by design — codex opt-in is similarly framed.** The "Want adversarial review? Run /codex:setup" line is correctly a hint, not a launch — good. Apply the same restraint to the `gh auth` offer (print the command, don't auto-launch the prompt).
- **Owner detection collides with worktrees.** `PWD == REPO_DIR` is correct in the original install.sh per `feedback_install_adopter_default_flip.md`, but if Justin uses `git worktree add ../mf-experiment` and runs install.sh from the worktree, `$PWD` will be the worktree path, not the canonical `~/Projects/MonsterFlow`. He'd be misclassified as adopter. Probably fine — he can pass `--install-theme` — but worth noting in the "Owner re-run" UX section.
- **Stakeholder not enumerated: cmux user with existing `~/.config/cmux/cmux.json`.** A cmux adopter who already has a tuned config will get prompted (adopter mode) or silently overwritten (owner mode). This is a real conflict — cmux is a relatively new tool, early-adopter overlap with MonsterFlow's audience is high, and these users are exactly the ones who'd already have a config.
- **Documentation parity acceptance is one-line and under-scoped.** "QUICKSTART.md updated to describe the new install/onboard split if any flow change affects users" — every change here affects users (theme prompt, panel output, onboard split). The "if" should be removed. README's one-liner working is necessary but not sufficient; the supporting prose around it needs review too.
- **Test harness uses temp `$HOME` — good — but doesn't simulate brew.** Acceptance case 4 invokes real `brew bundle install` against a real network. This makes the test slow, network-dependent, and unable to run in restricted CI. Consider a `BREW_DRY_RUN=1` mock or wrapper that asserts the invocation arguments without hitting Homebrew.

## Verdict

PASS WITH NOTES — the three-class stakeholder model (Owner, Adopter, CI) is solid and the existing institutional knowledge is correctly preserved, but the Linux-fail-mode, v0.4.x-upgrade-UX, and `gh auth login` non-TTY gaps each need a concrete answer before /plan can converge.
