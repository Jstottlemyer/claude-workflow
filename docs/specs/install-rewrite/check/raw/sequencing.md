# Sequencing Review — install-rewrite plan

## Verdict
PASS WITH NOTES — task ordering honors the three invariants and wave parallelism is sound, but a few sequencing details (W1 1.5/1.6 vs SIGINT, CHANGELOG source-of-truth direction, 2.5 owner-detect-before-persona-metrics) need a small clean-up before /build.

## Must Fix Before Building

- **Invariant 2 (SIGINT trap before first tmp write) is technically violated by W1 task 1.6.** Plan §D7 says `INSTALL_SCRATCH="$(mktemp -d -t monsterflow-install)"` is established by W2 task 2.2 ("SIGINT trap + `INSTALL_SCRATCH` mktemp"). But W1 task 1.6 ("source `python_pip` helper") is described as landing "After SIGINT trap, before migration detect" — yet 1.6 sits in W1 and 2.2 sits in W2. If commits land in stated wave order (W1 fully, then W2), then 1.6 will hit `install.sh` before 2.2, before the trap exists. Either (a) move 1.6 down into W2 between 2.2 and 2.3 (cleanest), or (b) note explicitly that 1.6's source-line is no-tmp-writing and therefore safe even before the trap is installed. The latter is true (sourcing a helper writes no files), but the plan's own placement note ("After SIGINT trap") is then misleading. Pick one.

- **CHANGELOG source-of-truth direction is inverted in W5 ship criterion.** W5 task 5.3 says "CHANGELOG.md migration bullets `diff` cleanly against install.sh's `print_upgrade_message` source." That treats install.sh as the source and CHANGELOG as derived. Conventionally CHANGELOG is the single source of truth for "what changed in v0.5.0" and code copies from it (or, ideally, sources a snippet from it). Two acceptable resolutions: (i) flip the dependency — write CHANGELOG.md first in W5, then `print_upgrade_message` is derived from / asserted against CHANGELOG; or (ii) explicitly document that install.sh's hardcoded message is the canonical 5-bullet diff and CHANGELOG.md restates it verbatim, with the `diff` test enforcing parity in either direction. Today's wording is ambiguous and will cause drift the first time someone updates one without the other.

## Should Fix

- **W2 task 2.5 (owner-detect swap) ordering relative to 2.7 (tier-split decline) and the existing persona-metrics block.** 2.5 is listed with `Depends On: 1.5` only, implying it can land any time after flag-parse. But the existing `OWNER` value drives the persona-metrics gitignore default-flip (install.sh:236-240) and the new theme stage (W2 2.8) owner-vs-adopter behavior. If 2.5 lands after 2.8 in the commit sequence, 2.8's mid-development behavior uses the old fragile `PWD == REPO_DIR` detection. Recommend pinning 2.5 BEFORE 2.8 explicitly (it already is in numeric order, but the `Depends On` column should say `1.5, before 2.8` to make the constraint a hard one for any reviewer reordering commits).

- **W2 commit count (10 sub-tasks 2.1→2.10) vs spec's "9 sub-tasks" claim in your prompt.** Plan §Implementation Tasks W2 lists 2.1 through 2.10 — that's 10, not 9. Ten commits at +20 lines avg = +200 lines, which is 10% above the +180 budget. Either accept the budget creep, merge two adjacent low-risk commits (e.g., 2.1 Linux guard + 2.4 Brew→REQUIRED are both single-line additions and could share a commit), or restate the budget to "+200/-8". Not blocking but worth picking before /build.

- **W3 vs W2 parallelism: env-var contract is the only coupling, and it's fine — but doctor.sh and bootstrap-graphify.sh are touched by W3.** W3 task 3.3 reads `bootstrap-graphify.sh` (already exists per spec §Touchpoints) and 3.1 invokes `doctor.sh`. If either of those scripts is mid-flight (open Q5 in plan recommends a 3-line defensive edit to `bootstrap-graphify.sh`), W3 needs to either land first or the defensive edit must be a discrete W5 follow-up. Plan already routes the defensive edit to W5 — confirm this is intentional and that W3 does not depend on the defensive behavior (it doesn't; the gating is via env vars W3 itself sets).

- **W4 deferral rationale ("test churn from in-flight behavior") is correct but partially mitigated by W2's `MONSTERFLOW_INSTALL_TEST` recursion-guard env var landing in 2.9.** W4 4.1 (harness skeleton) actually only depends on the env-var contract from 1.5 and the recursion-guard from 2.9 — it could start in parallel with 2.10 once 2.9 lands. The "wait for W2/W3 frozen" framing is conservative and probably correct (cases 1, 2, 3a will rewrite themselves if exact strings shift), but if calendar pressure mounts, 4.1 + 4.5 (negative cases — they don't depend on stage internals) could start one-day-earlier without harm.

## Observations

- **Three invariants validated.** Linux guard between current lines 2-4 (W2 2.1) lands before any path-computation that depends on macOS-specific binaries — correct. Migration detect (W2 2.3) lands after SIGINT trap (2.2) and before the first symlink mutation (current install.sh line 109+, all preserved-existing code that runs after W2 stages 2.4-2.8) — correct. SIGINT trap (2.2) lands before theme stage (2.8) which is the first new `.monsterflow.tmp` writer — correct, with the W1 1.6 caveat noted above.

- **W2 numeric order is compatible with dependency edges.** Walking 2.1→2.10: each task's `Depends On` references either an earlier W2 task or W1 1.5. No reverse edges, no cycles. The strict-sequential-commit framing in the plan is enforceable as written.

- **Wave 1 parallel-safety verified.** 1.1 (Brewfile) writes to `Brewfile`. 1.2/1.3/1.4 each write to a distinct new file in `config/`. 1.5/1.6 are sequential within W1 but neither touches what 1.1-1.4 touch. No hidden file-conflict between 1.1 and 1.5 (flag-parse doesn't reference cmux/tmux explicitly — the Brewfile is a downstream consumer, not a prerequisite for argv parsing). Genuinely parallel-safe.

- **W3 parallel to W2 once W1 closes — verified.** `scripts/onboard.sh` reads only `MONSTERFLOW_NON_INTERACTIVE` and `MONSTERFLOW_FORCE_ONBOARD` from the env, both established by W1 1.5. It does not source any function from install.sh, does not require install.sh to be at any particular stage of the W2 rewrite. The "depends only on env contract" framing in the plan §Architecture Summary holds.

- **Critical-path estimate.** Serial wall-clock: W1 (~30 min for 6 small tasks) → W2 (~3-4 hr for 10 commits with autorun-shell-reviewer between, since each is small but each needs shellcheck + manual smoke) → W4 (~2-3 hr for 4.1 harness + 4.2-4.5 cases) → W5 (~30 min docs). With W2 || W3 (W3 ~1 hr): total ~6-8 hr serial, ~5-6 hr with the one parallel band. Reasonable for one focused build session. The single biggest risk to landing the 10 W2 commits cleanly is the autorun-shell-reviewer subagent gate cited in the W2 ship criterion — if it surfaces a High finding mid-wave, the whole wave's commit chain may need a rebase. Plan accepts this risk implicitly; calling it out explicitly would help.

- **Commit boundaries: mostly right, two candidates for merge.** Each of 2.1-2.10 is a sensible standalone commit semantically. Two pairs are small enough to consider merging without losing review value: (a) 2.1 Linux guard (~3 lines) + 2.4 Brew→REQUIRED (~1 line) — both trivial additive lines in adjacent regions; (b) 2.9 NON_INTERACTIVE wrapping (~4 small wraps) + 2.10 onboard call (~3 lines) — both are "wire up the new flag/script into existing code." Merging would bring W2 from 10 to 8 commits and reclaim the +180 budget. Not a must-fix; standalone commits are fine if you prefer atomic auditability.

- **Net-line budget alignment.** Plan claims +180/-8, ~12 edited, ~530 final. With 10 W2 commits at the implied +20/commit average, the math works for W2 alone (200). But W1 also adds lines: config/ files (~40 lines for cmux.json + tmux.conf + zsh-prompt-colors.zsh combined), flag-parse block (1.5: ~30 lines for 7 flags + env exports + help text), python_pip source (1.6: ~3 lines). That's another ~73 lines of W1 net-add to install.sh + new files. The +180 budget is install.sh-only or repo-wide? If install.sh-only, W1 contributes only the ~33 lines from 1.5+1.6, W2 contributes ~150, total ~183 — fits the budget. If repo-wide, add ~40 for config/, ~80 for onboard.sh, ~150 for tests/test-install.sh = +450 repo-wide. Worth pinning which scope the +180 refers to so the post-merge audit doesn't surface a false overage.

- **"+9 W2 commits" in your check prompt vs "10 W2 commits" in plan.** Plan lists 2.1-2.10 (ten). Your prompt referenced "9 sub-tasks". Either the prompt was off-by-one or the plan grew between spec-revision and plan-finalize. Flagging so /build doesn't ship under the wrong target.
